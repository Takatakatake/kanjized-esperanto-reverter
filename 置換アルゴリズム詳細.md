# 置換アルゴリズムの詳細説明

## 概要

このアプリケーションは、漢字化エスペラントを元の小文字エスペラント語根へ戻すために、**最長一致（Longest Match）+ priority** のアルゴリズムを使います。

新しい `PEJVO/PIV 2026-06-20` 割当案では、同じ漢字に複数語根が割り当てられる場合、`藻ᶠᶜ` や `此ᶜ` のような上付き識別子つき表記が出ます。このため、逆変換では「上付き識別子を含む漢字側の文字列」を1つのキーとして扱います。

## 辞書の組み立て

CSV から以下の情報を読みます。

```csv
esperanto,kanji,priority
bon,良,0
fic,藻ᶠᶜ,0
bangi,藻ᴮ,0
```

内部では、まずエントリを次の順で並べます。

```python
entries.sort(key=lambda entry: (-len(entry.kanji), entry.priority, entry.source_order))
```

つまり:

1. `kanji` の長さが長いものを優先
2. 同じ長さなら `priority` が小さいものを優先
3. それでも同じなら CSV 上で先に出たものを優先

`PEJVO/PIV 2026-06-20` では、入力行数は 6,578、逆変換キー数は 6,577、最長キーは 6 文字です。

## 変換時の処理

入力テキストを左から右へ走査し、各位置で辞書に存在する長さを長い順に試します。

```python
for length in mapping.lengths_desc:
    substring = text[i : i + length]
    replacement = mapping.by_length[length].get(substring)
    if replacement is not None:
        result.append(replacement)
        i += length
        break
```

マッチしない文字はそのまま残します。最後に全体を小文字化します。

## 例

### 上付き識別子つき表記

入力:

```text
藻ᶠᶜoj
```

辞書:

```text
藻ᶠᶜ -> fic
藻ᴮ   -> bangi
藻    -> alg
```

結果:

```text
ficoj
```

短い `藻` を先に変換してしまうと `algᶠᶜoj` になってしまうため、上付き識別子を含む長いキーを先に見る必要があります。

### 同じキーが重複する場合

同じ漢字表記が複数回現れた場合は、priority が小さいものを採用します。

```csv
esperanto,kanji,priority
high,同,1
low,同,10
```

入力:

```text
同
```

結果:

```text
high
```

## 性能

時間計算量はおおむね `O(n * m)` です。

- `n`: 入力テキストの長さ
- `m`: 辞書に存在するキー長の種類数

実際にはキー長の種類は少ないため、通常の文章では十分軽く動きます。

## まとめ

1. 上付き識別子を含む漢字側の表記長で、長いものを先に戻す
2. 同じ長さでは漢字化優先順位、つまり `priority` の小さいものを先に戻す
3. マッチしない文字や既存のアルファベット部分はそのまま残す（ただし最後に出力全体を小文字化するため、大文字は保持されない）
